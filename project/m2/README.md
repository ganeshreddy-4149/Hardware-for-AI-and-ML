## Submitted Files and What They Do

All files are inside `project/m2/`. File names match the M2 checklist exactly.

| File | Path | What it does |
|------|------|--------------|
| `compute_core.sv` | `project/m2/rtl/compute_core.sv` | Synthesizable SystemVerilog RTL for the INT8 3x3 Conv2D compute core. Implements a 5-state FSM (IDLE→LOAD→COMPUTE→OUTPUT→DONE), a 9-element combinational MAC array with INT32 accumulator, a combinational requantizer (right-shift + clamp to INT8), and on-chip SRAM buffers for weights, activations, bias, and output. Single clock domain, active-high synchronous reset. |
| `interface.sv` | `project/m2/rtl/interface.sv` | Synthesizable SystemVerilog RTL for the AXI4-Lite slave interface. Implements a write FSM (IDLE→AW→W_DATA→W_RESP) and a read FSM (IDLE→AR→R_DATA) with full handshake on all five AXI4-Lite channels. Exposes four 32-bit registers: CTRL (0x00), STATUS (0x04), SHIFT (0x08), SCRATCH (0x0C). Drives core_start and core_shift to the compute core. |
| `tb_compute_core.sv` | `project/m2/tb/tb_compute_core.sv` | SystemVerilog testbench for compute_core. Loads a representative 3x3 Sobel-X test vector into the weight and activation buffers, runs the FSM through a full COMPUTE→OUTPUT cycle, captures result_out, and compares against an independently computed Python reference value of -6. Prints PASS or FAIL. Dumps compute_core.vcd for waveform viewing. |
| `tb_interface.sv` | `project/m2/tb/tb_interface.sv` | SystemVerilog testbench for interface. Drives four complete AXI4-Lite transactions: write CTRL=1 and verify core_start, read STATUS and verify busy/done bits, write SHIFT=5 and verify core_shift, read SHIFT back and verify roundtrip. All five AXI4-Lite channels exercised with full handshake. Prints PASS or FAIL. |
| `compute_core_run.log` | `project/m2/sim/compute_core_run.log` | Simulation transcript from running tb_compute_core.sv with Icarus Verilog 12.0. Shows buffer verification output and the final PASS line. Generated with: `vvp compute_core_sim 2>&1 \| tee compute_core_run.log` |
| `interface_run.log` | `project/m2/sim/interface_run.log` | Simulation transcript from running tb_interface.sv with Icarus Verilog 12.0. Shows all four AXI4-Lite CHECK lines and the final PASS line. Generated with: `vvp interface_sim 2>&1 \| tee interface_run.log` |
| `waveform.png` | `project/m2/sim/waveform.png` | Waveform screenshot showing compute_core signals: clk, rst, start, busy, done, result_valid, result_out. Generated by parsing compute_core.vcd with a Python matplotlib script. Shows the reset pulse, start assertion, busy high during COMPUTE, and result_out=-6 at the end of OUTPUT state. |
| `precision.md` | `project/m2/precision.md` | INT8 precision choice documentation. States the numerical format, justifies INT8 against the M1 roofline (AI=12.10 FLOP/byte), provides quantization error analysis across 100 samples (MAE=0.000974, max error=0.003494), compares against INT4 (19.83x worse), and states the acceptability threshold. 1042 words. |
| `README.md` | `project/m2/README.md` | This file. Lists all submitted files and what they do, gives exact command lines to reproduce both simulations, states the simulator version, and documents any deviations from the M1 plan. |

## Overview

This README documents how to reproduce all Milestone 2 simulations from a
clean clone of the repository. Both testbenches are written in SystemVerilog
and simulated using Icarus Verilog. No proprietary tools are required.

## Simulator

    Tool    : Icarus Verilog
    Version : 12.0 (stable) (v12_0)
    Install : built from source on Ubuntu 22.04 (Google Colab)

To install Icarus Verilog 12.0 from source on a fresh Ubuntu environment:

```bash
apt-get install -y build-essential autoconf gperf flex bison
git clone --depth=1 --branch v12-branch https://github.com/steveicarus/iverilog.git
cd iverilog
sh autoconf.sh
./configure
make -j2
make install
iverilog -V
```

Expected version output:

```
Icarus Verilog version 12.0 (stable) (v12_0)
```

## How to Run — Compute Core Testbench

### Step 1: Navigate to the repository root

```bash
cd your-repo
```

### Step 2: Check RTL syntax alone

```bash
iverilog -g2012 -o /dev/null project/m2/rtl/compute_core.sv
```

Expected output: nothing (zero errors).

### Step 3: Compile testbench and RTL together

```bash
iverilog -g2012 -o compute_core_sim \
    project/m2/tb/tb_compute_core.sv \
    project/m2/rtl/compute_core.sv
```

Expected output: nothing (zero errors).

### Step 4: Run the simulation

```bash
vvp compute_core_sim
```

Expected output:

```
VCD info: dumpfile compute_core.vcd opened for output.
=== Buffer verify ===
act [0,1,2]   = 1 2 3
act [32,33,34]= 4 5 6
act [64,65,66]= 7 8 9
wt  [0..8]    = 1 0 -1 1 0 -1 1 0 -1
mac_sum at h=0,w=0,oc=0 = -6  (expected -6)
=== Result ===
CHECK: result_out[oc=0,h=0,w=0]=-6 expected=-6  --> PASS
PASS
```

### Step 5: Save log

```bash
vvp compute_core_sim 2>&1 | tee project/m2/sim/compute_core_run.log
```

## How to Generate Waveform PNG

Both simulations dump a VCD file automatically when run.
To view the waveform:

1. Run either simulation as above — it creates `compute_core.vcd`
   or `interface.vcd` in the current directory.
2. Open the VCD in any waveform viewer. The waveform.png in
   `project/m2/sim/` was generated using a Python matplotlib
   script that parses the VCD and plots the signals.

Signals visible in waveform.png:
`clk`, `rst`, `start`, `busy`, `done`, `result_valid`, `result_out`

## Test Vector Description

### compute_core testbench

The testbench exercises the dominant kernel from M1 profiling: a 3x3
convolution with INT8 inputs and weights.

    Input activation patch (channel 0, rows 0-2, cols 0-2):
        [[1, 2, 3],
         [4, 5, 6],
         [7, 8, 9]]

    Weights (output channel 0, input channel 0, Sobel-X style):
        [[ 1,  0, -1],
         [ 1,  0, -1],
         [ 1,  0, -1]]

    Expected output (Python-verified):
        C[0][0] = 1*1 + 2*0 + 3*(-1) + 4*1 + 5*0 + 6*(-1) + 7*1 + 8*0 + 9*(-1)
                = 1 - 3 + 4 - 6 + 7 - 9 = -6

    Bias = 0, shift_amt = 0, no clamping needed.
    result_out at oc=0, h=0, w=0 = -6 ✓

### interface testbench

The testbench exercises all four AXI4-Lite register transactions:

    Test 1: Write 0x1 to CTRL (0x00)    → core_start must become 1
    Test 2: Read STATUS (0x04)           → rdata[1]=busy=1, rdata[0]=done=0
    Test 3: Write 0x5 to SHIFT (0x08)   → core_shift must become 5
    Test 4: Read SHIFT (0x08)            → rdata[4:0] must equal 5

All four AXI4-Lite channels are exercised with full handshake:
write address (AW), write data (W), write response (B),
read address (AR), read data (R).
