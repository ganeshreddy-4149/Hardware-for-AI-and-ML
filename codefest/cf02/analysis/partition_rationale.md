# HW/SW Partition Rationale

The kernel selected for hardware acceleration is a single-layer INT8 3x3 Conv2D used for CNN inference. This choice is supported by the profiling results, which show that `conv2d_int8_reference` is the dominant kernel in the current software baseline, accounting for about 88.09% of the total profiled runtime. The arithmetic intensity of the kernel is 12.1046 FLOP/byte, which makes it a reasonable candidate for custom acceleration compared with simpler memory-dominated operations. The roofline analysis therefore supports accelerating the Conv2D compute kernel rather than surrounding control code.

The software baseline will continue to handle host-side tasks such as input tensor preparation, loading weights and bias values, issuing the start/control sequence, collecting the output tensor, and comparing hardware results against the software reference model. In other words, software will manage orchestration and verification, while the hardware accelerator will perform the multiply-accumulate-heavy convolution operation itself.

At the hypothetical hardware design point, the target throughput is 3.2 GFLOP/s. Using the relation bandwidth = performance / arithmetic intensity, the required interface bandwidth is 0.2644 GB/s, or about 264.36 MB/s. For the interface assumption, I use AXI4-Stream with an FPGA SoC-style host, since the project documentation describes AXI4-Stream as a natural fit for inference pipelines with continuous data flow. Assuming a simple 32-bit AXI4-Stream link at 100 MHz, the available bandwidth is about 0.4000 GB/s, which is greater than the required 0.2644 GB/s. Under this assumption, the accelerator is not interface-bound at the chosen operating point.

On the current laptop CPU roofline, the Conv2D kernel is compute-bound in theory because its arithmetic intensity (12.1046 FLOP/byte) is greater than the ridge point (about 4.47 FLOP/byte). However, the measured software performance remains very low because the current implementation is a pure Python nested-loop reference model. The hardware design is expected to remain in the compute-bound region on the same roofline, but with much higher attainable performance due to a more efficient implementation of the same kernel.
